## #001 VKOSPI 수집 실패 (409 StockConflict)

| 항목 | 내용 |
|------|------|
| **현상** | `domestic.vkospi` 가 null, 리포트에 "미국 VIX (참고)" 표시 |
| **원인** | Naver 모바일 API (`m.stock.naver.com/api/index/VKOSPI/basic`) 가 상장지수에 VKOSPI를 포함하지 않아 409 반환. 08:00 KST 장전 시간에는 영구적으로 실패 |
| **해결** | 4단계 폴백: ① Naver sise_index HTML 파싱 → ② Naver mobile API (2회 재시도) → ③ 전일 `data.json` carry-forward → ④ Yahoo ^VIX |
| **예방** | 16:40 KST `supply_snapshot.js` 에서 VKOSPI를 수집해 `supply.json`에 저장 → 익일 STEP 2.5에서 병합 |
| **관련 파일** | `tools/collectors/domestic.js`, `tools/collectors/supply_snapshot.js`, `tools/main.js` (STEP 2.5) |
| **발생일** | 2026-05-09 ~ 2026-05-12 |

## #011 수급 데이터 소급 수집 불가 (Naver polling API 한계)

| 항목 | 내용 |
|------|------|
| **현상** | 특정 날짜(예: 5/12) 수급 데이터를 장 마감 후 다음날 소급 수집 시도 → 전부 실패 |
| **원인** | `polling.finance.naver.com/api/realtime/domestic/index/KOSPI_INVESTOR`는 **당일 실시간 API**. 어제 날짜 조회 파라미터가 없고, 다음날 장이 열리면 이전 데이터 접근 불가. Naver 금융 HTML 페이지는 수급 숫자를 JavaScript로 렌더링(서버 HTML에 없음). KRX 데이터포털은 브라우저 세션/CSRF 쿠키 필수 |
| **시도했던 방법** | ① Naver polling API → `datas:[]` ② Naver `investorDealTrendDay.naver` HTML 파싱 → 빈 결과 ③ KRX `getJsonData.cmd` POST → 400/403 ④ KRX OTP 방식 → 403 |
| **결론** | 수급 데이터는 **당일 16:30~16:40 사이에만** 수집 가능. 그 이후는 소급 불가 |
| **예방** | `supply-collect.yml`을 cron-job.org 외부 스케줄러로 정확히 16:40 KST에 실행. GA 크론만 믿으면 지연/누락 발생 → supply.json이 생성되지 않아 수급 카드 영구 미표시 |
| **현재 설정** | cron-job.org jobId: `7594700` — 16:40 KST 평일 자동 실행 |
| **관련 파일** | `tools/collectors/domestic.js` (`fetchKrxSupply`), `tools/collectors/supply_snapshot.js`, `.github/workflows/supply-collect.yml` |
| **발생일** | 2026-05-13 |

## 빠른 디버깅 체크리스트

리포트가 안 오면 이 순서로 확인:

```
1. GitHub Actions 실행 여부 확인
   → https://github.com/neofil0308-commits/daily-market-report/actions

2. 오늘 sent.flag 존재 여부 확인
   → outputs/YYYY-MM-DD/sent.flag

3. 로컬 수동 실행으로 직접 발송
   → node tools/main.js --now --dry-run
   → node tools/preview_send.js

4. 오류 로그 확인
   → GA Actions 탭 → 실패한 step 클릭
   → 로컬: 콘솔 출력 확인
```

## #016 디자인 에이전트 과다 위임으로 컨텍스트 단절

| 항목 | 내용 |
|------|------|
| **현상** | 같은 디자인 작업을 에이전트에게 여러 번 위임했지만 매번 이전 맥락을 모르는 상태에서 시작, 같은 실수 반복 (테이블 형식 → 카드 → 테이블 반복 등) |
| **원인** | 에이전트는 새 호출마다 컨텍스트가 초기화됨. 지시가 불충분하거나 제약 조건이 빠지면 이전에 해결한 문제가 다시 나타남 |
| **해결** | 에이전트 위임 시 프롬프트에 반드시 포함: ① 현재 코드의 정확한 위치(줄 번호) ② 이미 시도했고 실패한 방법 ③ 절대 바꾸면 안 되는 제약 ④ 목표 HTML 구조 명세(의사코드 수준) |
| **예방** | 핵심 제약(인라인 스타일만, table 태그 금지, CSS 변수 금지 등)은 에이전트 프롬프트 **맨 마지막**에 "기술 제약 (필수)" 섹션으로 항상 명시 |
